下面给你一份**按优先级的一次性修复路线图（仅配置，不改代码）**，目标是先把流程跑稳，再逐步恢复 RAGAnything。

## 先做“止血版”（30 分钟可执行）

### 1) 先只验证 LightRAG，避开 RAGAnything
你现在有两条独立故障链，先把 LightRAG 跑通：

```powershell
python examples/e2e_smoke_test.py --limit 1 --skip-raganything
```

### 2) 在当前终端临时提高超时并降并发（先不改文件）
```powershell
$env:LLM_TIMEOUT="900"
$env:EMBEDDING_TIMEOUT="120"
$env:MAX_ASYNC="1"
$env:LLM_TEMPERATURE="0.0"
$env:LLM_CONTEXT_LENGTH="32768"
$env:LLM_JSON_MODE="1"
```

然后分步跑（更容易定位）：
```powershell
python scripts/build_knowledge_graph.py --step index --parser mineru --engine lightrag --limit 1
python scripts/build_knowledge_graph.py --step extract --parser mineru --engine lightrag --limit 1
```

### 3) 通过标准（LightRAG）
- 日志中不再出现 `HealthCheckTimeoutError` / `Task forcefully terminated due to execution timeout`
- `index` 至少有 1 个成功文档
- `extract` 不再全部是 `Failed to parse JSON from LLM response`（允许少量失败）

---

## 再处理 RAGAnything（Windows 稳定优先）

你日志里 RAGAnything 的根因是 MinerU 自动选了 `lmdeploy-engine -> turbomind`，然后触发 `no kernel image`。  
先走稳定模式：**强制 MinerU 不走 turbomind GPU 路径**。

### 4) 临时强制 MinerU 后端与设备
```powershell
$env:RAGANYTHING_MINERU_BACKEND="pipeline"
$env:RAGANYTHING_MINERU_DEVICE="cpu"
```

单独验证：
```powershell
python scripts/build_knowledge_graph.py --step index --parser mineru --engine raganything --limit 1
```

### 5) 通过标准（RAGAnything）
- 不再出现 `[TM][FATAL] ... no kernel image is available for execution on the device`
- 不再出现 MinerU 的 `502 Bad Gateway`
- `index` 至少有 1 个文档成功

---

## 若你必须用 RAGAnything 的 GPU/VLM（性能优先）

这条建议直接走 **WSL Ubuntu 24.04 + CUDA 12.8/12.9 + 对应构建**，Windows 原生链路在你这套组合上不稳定概率高。  
在 Windows 原生上硬顶，会反复踩 `sm_120` / kernel image / backend 选择问题。

---

## 建议固化到 `.env`（稳定后再写入）

可把以下值写入 `/.env`（先临时验证通过再固化）：

```env
LLM_TIMEOUT=900
EMBEDDING_TIMEOUT=120
MAX_ASYNC=1
LLM_TEMPERATURE=0.0
LLM_CONTEXT_LENGTH=32768
LLM_JSON_MODE=1

RAGANYTHING_MINERU_BACKEND=pipeline
RAGANYTHING_MINERU_DEVICE=cpu
```
